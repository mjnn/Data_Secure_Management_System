# 前端页面对后端依赖说明（固化规范）

本文档规定：**每完成一个（或一组强关联的）前端业务页面/壳层能力后**，须编写对应的 **后端依赖清单**，便于后续接口联调与权限对齐。产品级契约仍以 **`docs/DSMS_IMPLEMENTATION_SPEC.md`** 为准；本清单侧重 **「这一页要调哪些 API、带什么参数、谁有权」**。

---

## 1. 何时编写 / 更新

| 时机 | 动作 |
|------|------|
| 新路由页面合并入 `main` / 准备提 PR 前 | **新增** `docs/frontend-backend-deps/<slug>.md`（或更新已有文件） |
| 同一页面增加 Tab、新弹窗且涉及 **新一类** 读写接口 | 在对应 `<slug>.md` 中 **追加小节** 或新表 |
| 仅样式/文案/纯前端交互变更 | **不必**改依赖清单；若交互隐含新接口意图，在清单中 **「待澄清」** 补一行 |

---

## 2. 文件放置与命名

- **目录**：`docs/frontend-backend-deps/`
- **命名**：与路由或页面主文件对应，**kebab-case**，例如：
  - `user-management.md` ← `UserManagementPage.vue`、`/user-management`
  - `project-management.md` ← `ProjectManagementPage.vue`、`/project-management`
  - `dashboard-shell.md` ← `DashboardPage.vue` 中与后端相关的顶栏/下拉等
- **索引**：同目录下 **`README.md`** 维护「页面 ↔ 清单文件 ↔ 源码路径」对照表。

---

## 3. 单页清单必备章节（模板）

复制以下结构到每个 `<slug>.md`，按实填写；无内容写 **「无 / 不适用」**。

```markdown
# <中文页面名> — 后端依赖清单

## 1. 前端入口
- **路由**：`path`、`name`（见 `frontend/src/router.js`）
- **主要组件**：`frontend/src/views/....vue`（及子组件路径）

## 2. 当前实现状态
- **已接真实 API**：列出方法与路径
- **Mock / sessionStorage / 仅前端**：逐项说明数据来源与持久化键名
- **角色与拦截**：页面级 `platform_role` / `is_superuser` 等（与 `usePortalMenuVisibility` 或页内逻辑一致）

## 3. 待对接 API（按能力块）

对每一块能力用表格（可多条）：

| 前端能力 | HTTP | 路径（相对 `/api/v1/...`） | Query / Body 要点 | 响应字段要点 | 权限（规格/实现约定） | 附录 A `behavior_key`（若有） |
|----------|------|------------------------------|-------------------|--------------|-------------------------|-------------------------------|

## 4. 与规格的差异 / 缺口
- 规格已写但前端未做：**列出**
- 前端已做但规格未写：**列出**并建议补规格或 README

## 5. 联调检查清单
- [ ] 列表分页 `skip`/`limit`、`total`/`items`
- [ ] 中文 `detail` / 错误提示
- [ ] 401/403 与全局刷新 token 行为
- [ ] 审计键与 OpenAPI `description` 一致（若该操作需审计）
```

---

## 4. 与 OpenAPI / 代码的对应关系

- 路径与方法以 **后端路由实现** 与 **`DSMS_IMPLEMENTATION_SPEC.md`** 为准；若不一致，**以规格为产品真源**，在 §「与规格的差异」中记录代码侧待修项。
- 若接口尚未实现，在表格 **「待对接 API」** 中标注 **「未实现」**，并指向规格章节号。
